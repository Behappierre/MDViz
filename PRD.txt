Product Requirements Document (PRD)
1) Product Overview
Product Name

MD Visualizer (working title)

Problem Statement

Users can read Markdown files in text editors, but non-technical users struggle to understand structure and reuse content with formatting intact. Existing tools either:

Render markdown but don’t support intuitive drag-and-drop organization, or

Offer rich text editing but break round-trip markdown fidelity.

Goal

Build a low-code-friendly web application where a user can:

Drag and drop a Markdown (.md) file

See a visual representation of the document

Reorder content blocks with drag-and-drop

Copy selected content with formatting preserved

Export the updated content back to valid Markdown

Success Criteria

95%+ structural fidelity for common markdown elements on import/export

Copy/paste preserves formatting in target apps (Google Docs, Word, Slack, Email) for core elements

Time-to-first-render under 1 second for files ≤ 500 KB

User can complete “drop → reorder → copy → export” flow in under 3 minutes without guidance

2) Objectives and Non-Objectives
Objectives (MVP)

Drag/drop .md file

Visual rendering of markdown blocks

Block-level drag-and-drop reorder

Copy selected block(s) as formatted content

Export edited document as markdown

Non-Objectives (MVP)

Real-time multi-user collaboration

Full desktop app/offline-first

Arbitrary embedded HTML editing

Git integration

Advanced publishing workflow

3) User Personas

Business User / Analyst

Wants to read markdown docs without syntax noise

Wants to paste formatted sections into reports/slides/emails

Product/Project Manager

Wants to reorganize docs quickly by dragging sections

Needs structure preserved for engineering handoff

Developer/Technical Writer

Wants round-trip markdown fidelity and predictable export

4) User Stories
Core User Stories

As a user, I can drag-and-drop a markdown file to load it.

As a user, I can view the markdown as visual blocks (headings, lists, tables, code).

As a user, I can drag blocks up/down to reorganize document flow.

As a user, I can copy selected block(s) and paste with formatting retained.

As a user, I can export/download the updated markdown file.

Secondary Stories

As a user, I can undo/redo actions.

As a user, I can see parsing errors for unsupported syntax.

As a user, I can preserve code blocks exactly during export.

5) Functional Requirements
FR-1 File Import

Accept file types: .md, .markdown, .txt

Drag/drop area + click-to-upload fallback

Max file size configurable (default 5 MB)

On upload, parse markdown into internal document model (AST)

Acceptance Criteria

Invalid file type rejected with clear message

Valid .md renders within 1 second for ≤ 500 KB file

FR-2 Markdown Parsing

Parse Markdown using CommonMark + GFM subset:

headings (h1-h6)

paragraphs

bold/italic/strikethrough

links

unordered/ordered/task lists

blockquotes

code blocks/inline code

horizontal rules

tables

Preserve unknown/unsupported nodes as raw blocks (non-destructive behavior)

Acceptance Criteria

Parsed document represented in canonical AST model

Unsupported syntax not discarded; round-trip safe mode enabled

FR-3 Visual Rendering

Render each top-level node as a “block card”

Support nested display for list children

Code blocks rendered monospaced with language label if present

Tables rendered with grid UI

Acceptance Criteria

All supported nodes visibly distinct and readable

Rendering order matches source markdown order

FR-4 Drag-and-Drop Reordering

User can drag block cards to new positions

Drag handles visible on hover/focus

Maintain structural constraints:

Do not allow illegal nesting via drag in MVP

Reorder only top-level blocks in MVP

Update internal AST upon drop

Acceptance Criteria

Reordered blocks persist in document model

Undo step created for each drop action

FR-5 Copy with Formatting

User can select one or multiple blocks

Copy action writes to clipboard:

text/html (formatted)

text/plain (fallback)

text/markdown (if supported by browser API path; otherwise append markdown to plain fallback metadata flow)

Preserve headings, lists, tables, links, inline styles, code formatting

Acceptance Criteria

Paste into Docs/Word/Slack retains core formatting

Plain-text fallback always available

FR-6 Export

Export button downloads updated .md

Filename default: original name + -edited.md

Markdown serialization from canonical AST

Acceptance Criteria

Export is valid markdown

Structure matches visual ordering and edits

FR-7 Undo/Redo (MVP-lite)

Undo/redo stack for:

reorder actions

inline text edits (if enabled in MVP)

Min stack depth: 50 actions

Acceptance Criteria

User can revert last action reliably with keyboard shortcuts (Ctrl/Cmd+Z, Shift+Ctrl/Cmd+Z)

FR-8 Error Handling

Parsing failures displayed with line reference where possible

Partial render mode for recoverable parse states

App should never hard-crash on malformed markdown

Acceptance Criteria

Malformed input returns user-friendly error state + retry option

FR-9 Accessibility

Keyboard accessible drag/reorder alternative (Move Up/Down buttons)

Focus states on interactive elements

Semantic headings and ARIA labels

Acceptance Criteria

Core flows are keyboard operable

Meets WCAG 2.1 AA for core interfaces

2) Proposed Stack (web-first)

Parser ecosystem:

remark-stringify

Drag/drop:

State:

ID generation:

(If your low-code platform doesn’t allow React libs, wrap this as embeddable JS module/web component.)

3) Canonical AST Strategy
Canonical Form

Use MDAST-like normalized JSON:

top-level root.children[]

each node gets stable nodeId

unknown nodes stored in raw with passthrough behavior

Why canonical?

Decouples parser/output libs from UI

Makes undo/redo and drag operations deterministic

Supports reliable serialization and testing

4) Parsing Pipeline

Read file as UTF-8

Parse markdown to AST

Normalize nodes:

attach nodeId

normalize whitespace policy

mark unsupported nodes

Store as canonical AST

Generate render model from AST

Error Recovery

If parser throws, attempt fallback parse mode

Surface line/column when available

Preserve original source for recovery/export

5) Rendering Engine Details

Render by node type mapping:

heading → HeadingBlock

paragraph → ParagraphBlock

list → ListBlock

table → TableBlock

code → CodeBlock

raw/unknown → RawBlock (read-only in MVP)

Virtualize long documents (post-MVP)

Block selection state separate from AST

6) Drag-and-Drop Details
MVP Rules

Reorder only top-level blocks

No cross-nesting transformations

Preserve children within block

Algorithm

On drag end:

sourceIndex, destinationIndex

immutable reorder in root.children

push previous AST snapshot pointer to undo stack

recompute positions

7) Clipboard Engine
Copy Flow

Given selected block IDs:

Extract nodes from canonical AST

Serialize selection to:

HTML fragment (sanitized)

markdown text

plain text (markdown stripped or normalized)

Write via Clipboard API as multi-MIME payload where available

Browser Fallback

If rich clipboard unsupported:

copy plain text

notify user “Rich format not supported in this browser context”

8) Export Engine

Serialize full canonical AST to markdown with deterministic formatting options:

list indentation style

code fence style

heading spacing

Normalize trailing newline

Download via blob URL

9) State Management

State domains:

docState: metadata, source, parse status

astState: canonical AST, version

uiState: selection, dragging, focus, warnings

historyState: undo/redo stacks

Undo/redo approach:

store patches (preferred) or bounded snapshots

bounded memory cap (e.g., 100 history entries)

10) Security & Trust Boundaries

Never inject unsanitized HTML into DOM

Sanitize HTML before render and before clipboard write

Disable script/event handler attributes

Escape unknown raw blocks in render mode

Consider Content Security Policy (CSP) in host app
